# 课程考核报告模板

## 📋 概述

本目录包含移动应用开发课程综合项目考核的四份周报模板，对应项目开发的四个阶段。这些模板基于《移动应用开发教师教学指导手册》中的考核实施指导图谱制定，并完整覆盖`02-技术栈图谱`中的所有主流技术栈。

**原生开发**：
- ✅ Android原生（Java/Kotlin）
- ✅ iOS原生（Swift）

**跨平台开发**：

**多端开发**：

## 🎯 使用说明

### 周报系列使用说明

**对于学生团队**：
1. **按周提交**：每周结束时填写对应周次的报告
2. **如实填写**：所有数据和描述必须真实准确
3. **完整填写**：尽可能填写所有相关内容，不要遗漏
4. **及时提交**：在规定时间内提交给指导教师

**对于指导教师**：
1. **过程监控**：通过周报了解项目进展和团队状态
2. **及时指导**：根据报告中的问题提供针对性指导
3. **质量把控**：确保项目按计划推进，质量达标
4. **公平评分**：依据报告和实际表现进行客观评分

**🔴 重要警告：四份报告缺一不可**

> **⚠️ 必须四个考核报告均提交，考核成绩才有效。缺少任何一份报告，大作业成绩为0分（占课程总成绩50%）！**

**后果说明**：
- ❌ 缺少1份报告：大作业成绩0分
- ❌ 缺少2份或以上：大作业成绩0分
- ❌ 迟交任何一份报告：按缺交处理，成绩0分

**考核报告1：演示答辩**
- **填写时间**：答辩前3天完成初稿，答辩当天提交最终版
- **格式选择**：
  - 可选MD格式（本文档）或PPT格式（自行制作）
  - 内容要求相同，任选其一
- **⚠️ 答辩现场操作要求**：
  - **录音**：答辩时学生自己实时录音
  - **计时**：学生自己控制时间（约13-15分钟）
  - **转文本**：答辩后立即将录音转成文字
  - **一致性**：录音和文本内容必须完全一致
- **重点内容**：
  - 三个必答题的详细回答（核心技术20分、架构设计30分、创新点40分）
  - 随机问答题（10分）
  - 视频演示文件（≤10分钟，MP4格式）
  - **答辩现场录音文件**（MP3格式，必须）
  - **录音转录文本**（Word或PDF，必须）
  - **个人负责项目源码压缩包**（必须）
  - **部署说明文档**（必须）
  - 个人负责的系统功能模块清单
- **评分占比**：100分，占课程总成绩12.5%（大作业50%的1/4）
- **提交方式**：电子版文档（MD或PPT） + 视频 + 录音 + 转录文本 + 源码压缩包 + 部署说明

**考核报告2：个人报告**
- **填写时间**：项目结束后3天内
- **🔴 重点考核指标（不规范为0分）**：
  - **系统核心类图（UML Class Diagram）** - 必须符合UML规范
  - **核心功能顺序图（UML Sequence Diagram）** - 必须符合UML规范
  - **系统架构图（Architecture Diagram）** - 必须体现分层结构
- **其他内容**：
  - 个人负责技术栈的深度介绍
  - 四个阶段的详细工作记录
  - 遇到的问题和解决方案
  - 代码示例和技术亮点
  - 个人成长和收获总结
  - 自评分数（0-100分）
- **独特要求**：
  - 必须包含个人电子签名
  - 必须提交三类技术图表（高清PNG/JPG）
  - 提交打印稿（签名版）
  - 个人贡献度必须与小组报告一致
- **评分占比**：100分，占课程总成绩12.5%（大作业50%的1/4）

  - 严禁擅自修改他人的个人贡献数据
  - 违反规定：**扣除本报告全部分值**

- ✅ 项目声明与签名

## ⚠️ 注意事项

### 填写要求

1. **真实性**：所有数据必须真实，不得虚构
2. **完整性**：尽可能填写所有相关项目
3. **准确性**：数据和描述要准确无误
4. **规范性**：使用规范的技术术语
5. **及时性**：按时提交，不得拖延

### 常见问题

**Q1：报告中的代码示例需要很详细吗？**
A1：不需要贴出所有代码，只需贴出关键核心代码片段，说明技术要点即可。

**Q2：如果某些功能没有完成怎么办？**
A2：如实填写完成情况，并在"问题与困难"部分说明原因和应对措施。

**Q3：各平台进度不一致怎么填写？**
A3：分别填写各平台的完成情况，说明差异原因，制定赶进度计划。

**Q4：中期检查没通过怎么办？**
A4：认真听取教师意见，在报告中制定改进计划，后续报告中跟踪改进效果。

**Q5：答辩时间超时或不足怎么办？**
A5：提前多次演练，严格控制时间，准备好备用方案应对意外情况。

## 💡 技术栈组合建议

### 推荐组合方案

**方案一：原生+跨平台**（适合深度学习）
- Android原生（Kotlin）
- iOS原生（Swift）
- Flutter（Dart）
- 后端服务（Node.js/Java/C#）

**方案二：跨平台为主**（适合快速开发）
- Flutter（Dart）
- React Native（TypeScript）
- Uniapp（Vue.js）
- 后端服务（Node.js/Python）

**方案三：多端全覆盖**（适合全栈能力）
- Android原生或Flutter
- Uniapp（覆盖小程序+H5）
- 微信小程序（深度定制）
- 后端服务（Java/Node.js）

**方案四：新技术探索**（适合创新项目）
- 鸿蒙ArkTS（多设备协同）
- Flutter（跨平台UI）
- 微信小程序（云开发）
- 后端服务（支持分布式）

### 技术栈搭配原则

1. **确保技术多样性**：至少包含2-3种不同类型的技术栈
2. **平衡深度与广度**：1-2个技术栈深入研究，其他了解基础
3. **考虑团队技能**：根据成员已有技能合理分配
4. **注重跨平台整合**：选择易于数据同步的技术组合
